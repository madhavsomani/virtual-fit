## Issues to Test (Prioritized)

### Issue 1: iOS Safari camera permission flow (HIGH)
**What:** iOS Safari shows the permission dialog differently than desktop Chrome. The dialog appears from the URL bar, not as a modal. Users may not see it if they're looking at the page content.
**Test:** Load `/mirror/` on iOS Safari → tap "Allow Camera & Start" → verify the pre-permission screen text is visible and the browser dialog appears in a way the user can find.
**Risk:** If the user taps "Don't Allow" once, Safari remembers and subsequent visits silently fail. Our error recovery screen says "tap the lock icon 🔒" — verify this instruction is accurate on current iOS.
**Fix opportunity:** Detect iOS Safari specifically and show "Go to Settings → Safari → Camera → Allow" as the recovery instruction.

### Issue 2: Camera aspect ratio mismatch on mobile (MEDIUM)
**What:** We request `{ width: { ideal: 640 }, height: { ideal: 480 } }` (4:3 landscape). Mobile front cameras often return 16:9 portrait (e.g., 1080×1920). The Three.js overlay canvas may not match the actual video dimensions, causing garment overlay misalignment.
**Code:** `server.js:1109` — `getUserMedia({ video: { facingMode, width: { ideal: 640 }, height: { ideal: 480 } } })`
**Test:** Load `/mirror/` in portrait mode on a phone. Check if the garment overlay aligns with shoulders or is offset/stretched.
**Fix opportunity:**
```javascript
// After stream starts, read ACTUAL dimensions
const vw = videoRef.current.videoWidth;
const vh = videoRef.current.videoHeight;
// Ensure Three.js canvas matches these exactly
```
Current code already does this (`videoDims.current = { w: vw, h: vh }`) but the Three.js camera ortho bounds may need updating when the video dimensions don't match the requested ones.

### Issue 3: Low-light / no-face-detected feedback loop (MEDIUM)
**What:** In low light, MediaPipe pose detection frequently fails (confidence < 0.4). The garment becomes invisible (`mesh.visible = false`) and the user sees nothing happening — they don't know if it's a bug or a lighting issue.
**Code:** `page.tsx:829` — `if (vis < 0.4) { mesh.visible = false; return; }`
**Test:** Cover phone camera partially or use in a dim room. Verify user gets feedback about lighting.
**Current behavior:** A `lowLightWarning` state exists (`line 133`) but it's unclear if it's wired to the pose visibility threshold.
**Fix opportunity:** When `vis < 0.4` for 3+ consecutive frames, show a toast: "💡 Move to a brighter area for better tracking"

### Issue 4: Embed widget iframe camera permissions (HIGH)
**What:** When VirtualFit loads inside an iframe (via `embed.js`), camera access requires the parent page to set `allow="camera"` on the iframe. If this attribute is missing, `getUserMedia` silently fails or shows a confusing error.
**Code:** `embed.js` sets `allow="camera *; microphone"` — but also sets `sandbox="allow-scripts allow-same-origin allow-camera"`. The `sandbox` attribute may BLOCK camera access on some browsers even with `allow-camera`.
**Test:** Load `/demo-store.html` on mobile. Tap "Try It On" → verify camera works inside the iframe panel.
**Fix opportunity:** Remove `sandbox` attribute from iframe (it's overly restrictive for a camera app), or add all needed sandbox flags.

### Issue 5: Portrait ↔ Landscape rotation handling (LOW)
**What:** Rotating the phone while camera is active may cause the Three.js canvas to not resize, leaving garment overlay at wrong dimensions.
**Code:** Orientation change listener exists at `line 416` but canvas resize may not trigger a Three.js renderer resize.
**Test:** Start camera in portrait → rotate to landscape → verify garment overlay still aligns.

## Quick Wins (no device needed)

### Fix A: Remove sandbox from embed iframe
The `sandbox` attribute in `embed.js` is likely breaking camera on iOS Safari embeds.

### Fix B: Add low-light feedback
Wire the existing `lowLightWarning` state to show when pose confidence stays below threshold.
